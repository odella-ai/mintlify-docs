# Odella Copywriting

Use this skill whenever creating or editing Odella copy. The core rule: describe Odella through a clear employment and workforce analogy, not technical AI infrastructure language.

## Mandatory terminology

### Preferred terms

Use these wherever possible:

- **AI Employee**
- **AI Employees**
- **AI workforce**
- **worker** / **workers**
- **team**
- **hire**
- **onboard**
- **train**
- **working hours**
- **role**
- **responsibilities**
- **tasks**
- **workflows**
- **work history**
- **performance**
- **manager / manage**
- **your team**

### Terms to avoid

Do not use these unless quoting an external source or a fixed API/entity name:

- **AI agent**
- **AI agents**
- **agent** as a noun for Odella’s product
- **operating hours**
- **runtime** in customer-facing copy
- **orchestration** in customer-facing copy
- **multi-agent system**
- **autonomous agent**
- **LLM pipeline**
- **model routing**
- **tool calling**
- **inference**
- **compute**
- overly technical implementation details

## Replacement rules

When editing existing copy, apply these replacements by default:

| Avoid | Use instead |
|---|---|
| AI agent | AI Employee |
| AI agents | AI Employees |
| agent | worker |
| agents | workers |
| operating hours | working hours |
| runtime | working time / active work time |
| deploy an agent | hire an AI Employee |
| configure an agent | onboard or train an AI Employee |
| agent marketplace | AI Employee marketplace / worker marketplace |
| agent builder | AI Employee builder |
| agent workflow | employee workflow / worker workflow |

Exception: keep `agent` only when it is part of a third-party product name, code/API terminology, file path, or unavoidable SEO keyword. If using it for SEO, prefer phrasing like: **“AI Employees, sometimes called AI agents…”** once, then return to Odella terminology.

## Voice and framing

Odella copy should be:

- Clear and practical
- Confident but not hype-heavy
- Common-language first
- Employment/workforce analogy first
- Outcome-focused
- Easy for non-technical buyers to understand

Avoid:

- Tech jargon
- Abstract platform language
- Overexplaining AI internals
- “Magic AI” claims
- Overpromising autonomy
- Saying Odella replaces people

## Product analogy

Explain Odella as a way to build and manage an AI workforce:

- Teams **hire** AI Employees.
- AI Employees have **roles** and **responsibilities**.
- They are **onboarded** with company context.
- They connect to workplace tools.
- They complete assigned tasks and workflows.
- They work during **working hours** or active work time.
- Humans manage, review, and improve their work.

## Examples

### Bad

“Deploy autonomous AI agents that orchestrate tool calls across your stack during operating hours.”

### Good

“Hire AI Employees that work across your tools, follow your workflows, and complete tasks during their working hours.”

### Bad

“Our AI agent marketplace includes prebuilt agents for sales, finance, and support.”

### Good

“Our AI Employee marketplace includes ready-to-hire workers for sales, finance, support, and operations.”

### Bad

“Configure agent runtime and model behavior.”

### Good

“Set each AI Employee’s role, responsibilities, tools, and working hours.”

## SEO guidance

Odella may still need to rank for market terms like “AI agent”, “AI agent builder”, or “AI agent marketplace”. When SEO requires those phrases:

1. Use Odella’s preferred term first.
2. Mention the market term secondarily.
3. Avoid repeating the market term more than necessary.

Example:

“Odella helps teams hire AI Employees — workplace-ready workers often described as AI agents — that connect to tools and complete business workflows.”

## Final review checklist

Before shipping Odella copy, check:

- [ ] Did we avoid “AI agent” unless required for SEO or third-party naming?
- [ ] Did we use “AI Employee”, “worker”, or “AI workforce” instead?
- [ ] Did we replace “operating hours” with “working hours”?
- [ ] Is the copy understandable to a non-technical buyer?
- [ ] Does it use employment/workforce analogies naturally?
- [ ] Does it avoid unnecessary technical implementation language?
- [ ] Does it avoid claims that AI Employees fully replace human teams?
